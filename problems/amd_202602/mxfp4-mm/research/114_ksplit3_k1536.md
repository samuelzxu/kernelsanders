# #114 KSPLIT=3 for K=1536 (get_splitk compatible)

## Discovery
get_splitk(768, 512, 3) → KSPLIT=3 PRESERVED! Verification:
- SPLITK_BLOCK_SIZE = ceil(2*ceil(768/3)/512)*512 = ceil(512/512)*512 = 512
- K_packed % (512//2) = 768 % 256 = 0 ✓
- 512 % 512 = 0 ✓
- K_packed % (512//2) = 768 % 256 = 0 ✓
- ACTUAL_KSPLIT = ceil(768/256) = 3 ✓

## Comparison
| Metric | KSPLIT=1 (current) | KSPLIT=3 (new) |
|--------|-------------------|---------------|
| Effective tiles | 192 | 576 |
| CU waves | 0.75 | 2.25 |
| BSK iters per tile | 3 | 1 |
| Reduction kernel | No | Yes (3-way, 768 blocks) |

Key difference from #108: BSK stays at 512 (same per-iteration compute),
only KSPLIT changes. Each tile does fewer iterations but more tiles saturate CUs.

## Results
WORSE: M=256 K=1536 → 26.7µs (was 20.0µs, +6.7µs)
The 3-way reduction kernel overhead (768 blocks, ACTUAL_KSPLIT=3) dominates.
1 BSK iteration per tile is too little work to amortize tile setup.

CONFIRMED: For K=1536, KSPLIT=1 with 3 BSK iterations is optimal despite
only 0.75 CU utilization. Large tiles with more work per tile > more small tiles.
