# #156 - waves_per_eu=2 for K=2048 + GROUP_SIZE_M=4

## Hypothesis

The LLVM O1 patch (#148) reduces register pressure vs O3, creating headroom
for higher `waves_per_eu` without register file conflicts.

`waves_per_eu` tells the hardware how many wavefronts to target per EU
simultaneously. When one wavefront stalls on memory, the hardware switches
to another. For K=2048 shapes that have significant HBM traffic (7.4MB of
B_q + A), higher waves_per_eu can hide this latency.

Current K=2048 configs use waves_per_eu=1. With O1:
- Fewer registers used → more wavefronts can coexist per EU
- waves_per_eu=2 doubles the in-flight work per EU

Also tests GROUP_SIZE_M=4 for the "any" K=2048 config as a midpoint between
the current GROUP_SIZE_M=2 and the 8-XCD-aligned GROUP_SIZE_M=8 from #153.
For N=7168, BLOCK_N=128: GROUP_SIZE_M=4 → 4*56=224 tiles per group = 28/XCD.

## Changes from #148

- K=2048 M_LEQ_64: waves_per_eu 1 → 2
- K=2048 "any": waves_per_eu 1 → 2, GROUP_SIZE_M 2 → 4
- All other shapes: unchanged
