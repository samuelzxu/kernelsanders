# #157 - Balanced KSPLIT=7 + BLOCK_K=512 for K=7168

## Problem with Current KSPLIT=8

K=7168 with BLOCK_K=256 gives 28 total K-blocks (7168/256=28).
KSPLIT=8: 28/8 = **3.5 (NOT integer)**. Each CTA nominally handles 896 K elements, but with BLOCK_K=256 that's 3 full iterations + 1 partial (128/256 elements). The partial last block:
- Executes BLOCK_K=256 MFMA instructions but only 128 are valid
- Wastes 50% of that block's MFMA work
- 1 partial block per 4 = **12.5% overall MFMA waste**

KSPLIT=7: 28/7 = **4.0 (exactly integer)**. Zero wasted work.

## BLOCK_K Upgrade: 256 → 512

IREE issue #22309 identified that BLOCK_K=256 with 16×16 MFMA instructions
(128 K-elements per instruction) gives only 2 MFMAs per K-iteration — too few
for adequate pipelining on gfx950. BLOCK_K=512 = 4 MFMAs per iteration,
providing better overlap between memory loads and compute.

With BLOCK_K=512, KSPLIT=7: K_per_split = 7168/7 = 1024. 1024/512 = **2 iterations** (exactly integer). 4 MFMAs per iteration.

## num_stages: 3 → 2

With only 2 K-iterations per CTA, 3 pipeline stages means the 3rd prefetch
buffer is never used (no K[iter+2] to prefetch). num_stages=2 is sufficient
and saves LDS bandwidth/space for higher occupancy.

## Expected Impact

- k=7168, m=16, n=2112 ranked shape: 12.5% MFMA efficiency gain + better
  pipeline overlap should reduce this shape from ~21.6µs to ~18-19µs.
- Geomean improvement: ~0.3-0.5µs improvement.

## Changes from #148

- K=7168 M_LEQ_16: KSPLIT 8→7, BLOCK_K 256→512, num_stages 3→2
- All other shapes: identical to #148
