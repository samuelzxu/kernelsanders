# Attempt 143: a16w8 for kv>1024 (skip Q quantization)

## Key Insight
The Q quantization step (Triton dynamic_per_tensor_quant_fp8_i8) adds:
- Kernel launch overhead: ~2µs
- Memory: read bf16 Q + write fp8 Q (tiny: bs=256 Q = 4.5MB bf16 → 2.25MB fp8)
- Total: ~3µs per kv>1024 call

Using a16w8 (bf16 Q + fp8 KV) for kv>1024 eliminates this overhead.

## Why it's safe
- Q data is tiny vs KV: bs=256 Q = 4.5MB bf16, KV = 1.2GB fp8 (267:1 ratio)
- Extra bf16 Q memory read: 4.5MB → 0.85µs at 5.3TB/s (negligible)
- a16w8 kernel handles variable kv lengths (same as a8w8)
- kvg=64 and nks=32 unchanged

## Expected Impact
- bs=32/kv=8192: 88.6 → ~86µs (-3µs Q quant)
- bs=64/kv=8192: 141 → ~138µs (-3µs)
- bs=256/kv=8192: 338 → ~335µs (-3µs)
- Geomean: 76.7 → ~75µs

Small but real improvement. Low risk.

## Also: nks=16 for kv<=1024
- Current: nks=8 → bs=64: 2 waves, bs=256: 8 waves
- With nks=16: bs=64: 4 waves, bs=256: 16 waves
- More waves = better HBM latency hiding (validated: nks=32 > nks=8 for kv>1024)
- Risk: larger reduce buffers (2x), but parallelism benefit should dominate
