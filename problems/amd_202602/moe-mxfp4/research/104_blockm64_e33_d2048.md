# v104: block_m=64 Override for E=33 bs=512 d=2048

## Problem
E=33 bs=512 d=2048 takes ~353µs with default CK 2-stage.
Default: block_m=128 → only 26% CU utilization (66/256 M-blocks).

## Shape Analysis
- E=33, bs=512, d_expert=2048, d_hidden=7168, topk=9
- tokens_per_expert = 512*9/33 = 139.6
- Default block_m=128 → ceil(139.6/128) = 2 M-blocks/expert → 66 total M-blocks
- 66/256 = 26% CU utilization (terrible!)

## Key Insight
With block_m=64:
- ceil(139.6/64) = 3 M-blocks/expert → 99 total M-blocks
- 99/256 = 39% CU utilization (still low but 50% better than 128!)

## Why Not block_m=32?
v54 showed block_m=32 for d=2048 was 21% SLOWER (413µs vs 340µs).
block_m=32 makes each tile too small for efficient FP4 MFMA pipeline utilization.
block_m=64 may find a better trade-off: more waves vs per-tile efficiency.

## Cannot Use cktile
inter_dim=2048 > 1024, so cktile path is excluded (BF16 MFMA too slow for large K).

## Implementation
Simply override md.block_m = 64 for the case:
- inter_dim > 1024 (NOT cktile)
- expert <= 33 (small number of experts, low CU utilization)
- tokens_per_expert >= 100 (dense enough that CU utilization matters)

## Expected Impact
E=33 bs=512 d=2048: potentially 5-15% faster from better CU utilization
Risk: if FP4 MFMA needs large M for efficiency, this might be slower
