# v106: cktile for E=33 d=2048 (high-risk, high-reward)

## Hypothesis
Remove the inter_dim>1024 exclusion for cktile to cover E=33 bs=512 d=2048.

## CU Utilization Analysis
E=33 bs=512 d=2048: tokens_per_expert=139.6

### Current (default CK):
- block_m=128: ceil(139.6/128)=2 per expert → 66 total → 66/256=26% CU util

### With cktile block_m=16:
- ceil(139.6/16)=9 per expert → 297 total → 297/256=116% CU util (FULL!)
- 4.5x more M-blocks = 4.5x more parallelism

## Risk Analysis
- FP4×FP4 MFMA: ~2048 TFLOPS peak (MI355X)
- BF16×FP4 MFMA: ~512 TFLOPS peak (4x lower)
- Net: 4.5x occupancy / 4x lower throughput ≈ +10% speedup if theory holds

## Why This Might Work
The 353µs baseline is bottlenecked by poor CU utilization (26%).
If cktile allows filling all 256 CUs, even at lower MFMA throughput,
total time could improve significantly.
Plus: cktile avoids FP4 quant/requant (saves ~5-10µs).

## Why This Might Not Work
- BF16 MFMA 4x lower throughput for K=7168 (stage1) dominates
- Large K dimension means memory bandwidth also higher (BF16 vs FP4 activations)
- cktile overhead (kernel launch, split-k reduction if enabled)

## Implementation
Simply extended the cktile condition to cover all E=33 shapes:
```python
elif expert <= 33 and tokens_per_expert < 150:
    use_cktile = True; sk = 1
```
No longer gates on inter_dim <= 512.
